## Learning Objectives

- Understand why Anthropic's Claude Code team moved from Markdown to HTML for human-facing output.
- Identify specific HTML features that improve readability, engagement, and decision-making for humans.
- Recognise potential costs and how teams are mitigating them.
- Apply these insights to your own agent system design.

## Prerequisites

You should be comfortable reading and writing both Markdown and HTML, and you should have seen how an AI agent presents code, plans, or research to a user.

## Content

### The Problem with Markdown for Complex Outputs

Markdown is excellent for lightweight, linear documentation. But as AI agents produce longer, more complex outputs — code reviews, system plans, multi-step research — Markdown's "wall of text" becomes a liability:

- **Low information density** — plain headings and lists fail to surface relationships or highlight changes.
- **No interactivity** — users cannot expand details, switch tabs, or see diagrams inline.
- **Cognitive overload** — reading a 500-line Markdown plan forces the human to hold context entirely in their head.

### What HTML Unlocks

By emitting HTML, an agent can treat its output as a **mini-application** rather than a plain document:

| Capability | Markdown | HTML |
|------------|----------|------|
| Colour-coded diffs | relies on syntax-highlighted code blocks | inline `<span>` styling, background colours |
| Tabbed interfaces | not supported | multiple views without scroll-hell |
| Inline SVG diagrams | only links to images | `<svg>` elements embedded and rendered |
| Interactive sliders / input | not supported | `<input type="range">`, JavaScript |
| Responsive tables | basic tables, no styling | CSS styled, scrollable on mobile |
| Two-way interaction | not supported | buttons, forms, real-time updates |

### Real-World Use in Claude Code

Claude Code now generates HTML artifacts for:

- **Code reviews** — colour-coded diffs with side-by-side before/after.
- **Execution plans** — tabs for different user journeys, each with a dynamically generated flow diagram (SVG).
- **Research summaries** — expandable sections, inline citations that pop up on hover.

### Trade-Offs and Mitigations

| Concern | Mitigation |
|---------|------------|
| **Token overhead** | Stream only the content the user currently sees; keep semantic structure to avoid redundant `<div>` soup. |
| **Diff readability** | Treat HTML as a **compiled artifact**; store source logic in Markdown/JSON and regenerate on change. Diff the source, not the HTML. |
| **Security** | Sanitise all user-injected content; use Content Security Policy (CSP); isolate agent-generated HTML in a sandboxed iframe. |

### When to Use HTML vs Markdown

A practical rule for agent designers:

- **Use HTML** when the human needs to interact with the output (filter, expand, compare) or when visual cues significantly aid comprehension.
- **Use Markdown** for quick, linear responses — simple explanations, command-line help, or outputs that feed directly into another machine process.
- **Default to Markdown** for agent-to-agent messages (control plane) and **HTML** for agent-to-human presentations (presentation plane).

## Exercises

1. Take a Markdown code-review report and re-design it as an interactive HTML page with colour-coded diffs. What new information can you convey?
2. For a system you're building, define a complexity threshold: "If the output is longer than N lines or contains more than M distinct sections, switch to HTML."
3. Audit an HTML output generated by an AI agent for security: list every place user-supplied data could end up inside a `<script>` tag or an event handler. Propose sanitisation steps.
